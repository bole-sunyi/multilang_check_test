# 多语言 UI 检查辅助工具需求文档

## 1. 项目定位

本项目是一个面向移动游戏活动页面的多语言 UI 检查辅助工具。工具通过 Airtest + Poco 连接 Android 模拟器或真机，按 YAML 配置执行业务路径，自动采集关键页面截图，写入多语言测试 Excel 模板，并生成 Markdown、HTML、JSON 报告，供测试人员和业务同学人工核对多语言 UI 展示。

当前项目的核心目标不是自动判断翻译语义是否正确，而是稳定、低成本地完成“进入页面、截图留证、整理到模板、输出报告”的检查链路。

## 2. 使用角色

- 测试人员：维护 `config/*.yaml`，执行指定活动模块，查看截图和 Excel 结果。
- 多语言检查人员：打开 Excel 模板，按步骤核对不同语言下的 UI 文案、排版、遮挡、截断和按钮状态。
- 自动化维护人员：维护公共执行器、Poco selector、报告和辅助采集工具。
- 新接手成员：通过 README 和本需求文档理解项目边界、运行方式和优化方向。

## 3. 业务范围

### 3.1 当前支持模块

项目当前保留 3 个独立业务模块：

- `stamp_test`：Stamp 活动链路截图检查。
- `byd_test`：BYD 活动链路截图检查。
- `atw_test`：ATW 活动链路截图检查。

每个模块独立运行，不要求一条命令连续执行全部模块。这样可以降低单次调试成本，也避免某个模块失败影响其它模块产物。

### 3.2 非目标范围

- 不在当前阶段实现完整 OCR 自动判定。
- 不在当前阶段自动判断翻译语义准确性。
- 不强制替代人工多语言检查流程。
- 不把旧批量回归入口作为多语言截图主流程。

## 4. 核心功能需求

### 4.1 单模块执行

- 工具应支持直接执行 `cases/stamp_test.air/stamp_test.py`、`cases/byd_test.air/byd_test.py`、`cases/atw_test.air/atw_test.py`。
- 每个入口脚本应复用统一执行器，避免模块之间复制公共逻辑。
- 执行器应负责连接设备、启动 App、读取 YAML、初始化 Poco、执行步骤、截图、写 Excel 和生成报告。
- 默认设备为 `127.0.0.1:16448`，可通过 `DEVICE_SERIAL` 覆盖。
- 默认包名为 `slots.pcg.casino.games.free.android`，可通过 `APP_PACKAGE_NAME` 覆盖。
- 默认产物目录为 `/Users/sunyi/Downloads/artifacts`，可通过 `ARTIFACTS_ROOT` 覆盖。

### 4.2 YAML 配置驱动

- 每个模块应有独立 YAML 配置文件，路径为 `config/{module_name}.yaml`。
- YAML 应包含基础配置，例如启动等待时间、是否停止 App、是否导出 Poco 控件树、Excel sheet 名。
- YAML 应包含 `steps` 列表，用于描述业务操作步骤。
- 步骤应支持以下动作：
  - `sleep`：等待页面、动画或资源加载。
  - `click`：点击控件或兜底坐标。
  - `snapshot`：主动截图留证。
  - `wait`：等待某个 Poco 控件出现。
  - `assert_exists`：断言某个 Poco 控件存在。
- `click` 应优先使用 `selector`，当 Poco 无法识别游戏贴图控件时再使用 `fallback_pos`。
- `fallback_pos`、`relative_pos` 坐标必须是 0 到 1 之间的二维归一化坐标。
- 配置错误应在执行前尽早报出清晰错误，而不是执行到中途才出现难以理解的异常。

### 4.3 截图采集

- 工具应按步骤顺序保存截图。
- 截图文件名应包含固定序号和步骤名称，便于按文件名排序。
- 截图应统一修正为横屏方向，保证报告和 Excel 中展示一致。
- `snapshot_after` 为 true 的步骤应在动作执行后自动补充截图。
- 点击后截图前应支持等待时间，避免截到动画中间态。

### 4.4 Excel 输出

- 默认写入 `/Users/sunyi/Downloads/多语测试模板.xlsx`。
- 支持通过 `MULTILANG_EXCEL_TEMPLATE_PATH` 或 `EXCEL_TEMPLATE_PATH` 覆盖模板路径。
- 每个模块应写入独立 sheet：
  - `stamp_test` 写入 `stamp`。
  - `byd_test` 写入 `byd`。
  - `atw_test` 写入 `atw`。
- Excel 应从第 3 行开始写入截图记录，保留模板表头。
- 输出内容至少包含步骤/截图名称、截图图片、定位方式。
- 重复执行同一模块时，应清理旧图片和旧行内容，避免新旧结果叠加。

### 4.5 报告输出

- 每个模块执行后应生成 `report.json`、`report.md`、`report.html`。
- 报告应包含执行状态、设备、模块名、日志目录、错误信息和截图记录。
- 报告应标识本次截图总数，以及使用 `fallback_pos` 的步骤数量。
- 报告应列出使用坐标兜底的截图步骤，提醒后续优先替换为 Poco selector。
- HTML 报告应提供本地文件链接，便于快速打开日志、截图和错误文件。

### 4.6 辅助采集工具

- `quick_click.py` 应用于连续采集真实点击坐标，并生成可复制到 YAML 的片段。
- `dump_current_screen.py` 应用于导出当前页面截图和 Poco 节点树。
- `poco_hover_inspector.py` 应用于在截图上悬停查看归一化坐标和节点信息。
- 辅助工具产物应统一写入 `ARTIFACTS_ROOT` 下的子目录，避免污染项目根目录。

### 4.7 旧批量回归能力

- `run_regression.sh` 可继续作为旧批量回归入口保留。
- 旧入口可支持设备配置、多设备并行、重试、基线 diff 和自动创建缺失 baseline。
- README 中应明确旧批量回归不是当前多语言截图主入口。

## 5. 非功能需求

- 易维护：日常维护应尽量只改 YAML，不改 Python 源码。
- 易排障：每次执行应保留截图、Poco 节点树、日志和报告。
- 稳定性：点击动作优先使用 Poco selector，坐标只作为临时兜底。
- 可读性：报告和 Excel 应面向非开发人员可读。
- 可迁移：设备、包名、产物目录、Excel 模板路径应支持环境变量覆盖。
- 低侵入：产物默认写到下载目录，避免把项目目录塞满。

## 6. 优化需求

### 6.1 已纳入本轮优化

- 增加 YAML 配置校验，提前发现空步骤、非法动作、非法坐标和缺失定位信息。
- 在报告中增加多语言截图采集摘要。
- 在报告中突出 `fallback_pos` 使用情况，辅助后续 selector 改造。
- 将悬停检查工具的临时截图移动到 artifacts 子目录。

### 6.2 后续建议

- 为 YAML 增加独立 lint 命令，不连接设备也能检查配置。
- 为新增模块提供脚手架，自动生成入口脚本和 YAML 模板。
- 建立模块注册机制，减少新增模块时手动复制文件。
- 建立 selector 覆盖率目标，例如每个模块坐标兜底步骤逐步降低到 30% 以下。
- 给 Excel 增加执行时间、设备号、模块名和错误摘要。
- 为 YAML 加载、Excel 导出和报告生成补充单元测试。
- 后续可按风险引入 OCR 文案识别，但应先保证截图链路稳定。

## 7. 验收标准

- 任一模块可通过对应入口脚本独立执行。
- 配置错误能在执行前输出清晰提示。
- 执行成功后，产物目录中存在截图、报告和 Poco 节点树。
- Excel 对应 sheet 中能看到按步骤排序的截图记录。
- 报告能展示截图总数和 `fallback_pos` 风险步骤。
- 辅助工具不会在项目根目录遗留临时截图。
