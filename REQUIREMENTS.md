# 多语言 UI 检查辅助工具需求文档

## 1. 项目定位

本项目是一个面向移动游戏活动页面的多语言 UI 检查辅助工具。工具通过 Airtest + Poco 连接 Android 模拟器或真机，按 YAML 配置执行业务路径，自动采集关键页面截图，生成钉钉 MCP 待同步数据和 HTML 报告，供测试人员和业务同学人工核对多语言 UI 展示。

当前项目的核心目标不是自动判断翻译语义是否正确，而是稳定、低成本地完成“进入页面、截图留证、整理到模板、输出报告”的检查链路。

## 2. 使用角色

- 测试人员：维护 `config/*.yaml`，执行指定活动模块，查看截图、钉钉在线表格和 HTML 报告。
- 多语言检查人员：打开钉钉在线表格，按步骤核对不同语言下的 UI 文案、排版、遮挡、截断和按钮状态。
- 自动化维护人员：维护公共执行器、Poco selector、报告和辅助采集工具。
- 新接手成员：通过 README 和本需求文档理解项目边界、运行方式和优化方向。

## 3. 业务范围

### 3.1 当前支持模块

项目当前保留 3 个独立业务模块：

- `stamp_test`：Stamp 活动链路截图检查。
- `byd_test`：BYD 活动链路截图检查。
- `atw_test`：ATW 活动链路截图检查。

每个模块独立运行，不要求一条命令连续执行全部模块。这样可以降低单次调试成本，也避免某个模块失败影响其它模块产物。

### 3.2 非目标范围

- 不在当前阶段实现完整 OCR 自动判定。
- 不在当前阶段自动判断翻译语义准确性。
- 不强制替代人工多语言检查流程。
- 不把旧批量回归入口作为多语言截图主流程。

## 4. 核心功能需求

### 4.1 单模块执行

- 工具应支持直接执行 `cases/stamp_test.air/stamp_test.py`、`cases/byd_test.air/byd_test.py`、`cases/atw_test.air/atw_test.py`。
- 每个入口脚本应复用统一执行器，避免模块之间复制公共逻辑。
- 执行器应负责连接设备、启动 App、读取 YAML、初始化 Poco、执行步骤、截图、生成钉钉 MCP 待同步数据和生成报告。
- 单模块启动后应自动发现可连接的 adb 设备/模拟器端口；多台设备时允许用户选择，也可通过 `DEVICE_SERIAL` 覆盖。
- 默认包名为 `slots.pcg.casino.games.free.android`，可通过 `APP_PACKAGE_NAME` 覆盖。
- 默认产物目录为 `/Users/sunyi/Downloads/multilang_check_artifacts`，可通过 `ARTIFACTS_ROOT` 覆盖。

### 4.2 YAML 配置驱动

- 每个模块应有独立 YAML 配置文件，路径为 `config/{module_name}.yaml`。
- YAML 应包含基础配置，例如启动等待时间、是否停止 App、是否导出 Poco 控件树、表格 sheet 名。
- YAML 应包含 `steps` 列表，用于描述业务操作步骤。
- 步骤应支持以下动作：
  - `sleep`：等待页面、动画或资源加载。
  - `click`：通过 Poco selector 点击控件。
  - `snapshot`：主动截图留证。
  - `wait`：等待某个 Poco 控件出现。
  - `assert_exists`：断言某个 Poco 控件存在。
- `click` 必须使用游戏 `StdPoco` selector；游戏节点推荐使用 `name + chain` 定位。
- `relative_pos` 坐标必须是 0 到 1 之间的二维归一化坐标。
- 配置错误应在执行前尽早报出清晰错误，而不是执行到中途才出现难以理解的异常。

### 4.3 截图采集

- 工具应按步骤顺序保存截图。
- 截图文件名应包含固定序号和步骤名称，便于按文件名排序。
- 截图应统一修正为横屏方向，保证报告和钉钉表格中展示一致。
- `snapshot_after` 为 true 的步骤应在动作执行后自动补充截图。
- 点击后截图前应支持等待时间，避免截到动画中间态。

### 4.4 钉钉 MCP 待同步输出

- 当前默认使用方案 2（Cursor MCP 版）：本地脚本生成待同步 JSON，由 Cursor MCP 写入钉钉在线表格。
- 钉钉结果表使用用户模板：`https://alidocs.dingtalk.com/i/nodes/dpYLaezmVNLd17qQSPXQzPAq8rMqPxX6?utm_scene=team_space`。
- 每条截图记录只写入前两列：
  - A 列 `描述`：YAML 中的 `name`；
  - B 列 `截图`：Cursor MCP 上传后的真实截图图片。
- 如果关闭 `dingtalk_export.enabled`，才回退到本地 Excel 输出。
- 本地脚本不得要求用户提供钉钉 `appKey`、`appSecret`、`accessToken` 或 `operatorId`。
- Cursor MCP 写入时应从钉钉表格现有内容后追加，不覆盖历史记录。

### 4.5 报告输出

- 每个模块执行后最终只保留 HTML 报告。
- 报告应包含执行状态、设备、模块名、错误信息和截图记录。
- 报告应标识本次截图总数和每张截图对应的 Poco selector 摘要。
- HTML 报告应提供本地文件链接，便于快速打开截图和报告关联文件。

### 4.6 辅助采集工具

- `dump_current_screen.py` 应用于导出当前页面截图、Poco 节点树和可复制执行的 `name + chain` YAML 片段。
- `poco_hover_inspector.py` 应用于在截图上悬停查看归一化坐标和节点信息。
- 辅助工具产物应统一写入 `ARTIFACTS_ROOT` 下的子目录，避免污染项目根目录。

### 4.7 旧批量回归能力

- `run_regression.sh` 可继续作为旧批量回归入口保留。
- 旧入口可支持设备配置、多设备并行、重试、基线 diff 和自动创建缺失 baseline。
- README 中应明确旧批量回归不是当前多语言截图主入口。

## 5. 非功能需求

- 易维护：日常维护应尽量只改 YAML，不改 Python 源码。
- 易排障：每次执行只保留表格、截图目录和报告文件，其它临时日志或节点树执行结束后清理。
- 稳定性：点击动作只使用 Poco selector，不使用坐标兜底。
- 可读性：报告和钉钉在线表格应面向非开发人员可读。
- 可迁移：设备、包名、产物目录、钉钉表格目标应支持配置或环境变量覆盖。
- 低侵入：产物默认写到下载目录，避免把项目目录塞满。

## 6. 优化需求

### 6.1 已纳入本轮优化

- 增加 YAML 配置校验，提前发现空步骤、非法动作、非法坐标和缺失定位信息。
- 在报告中增加多语言截图采集摘要。
- 自动生成和模块执行统一使用 Poco selector。
- 将悬停检查工具的临时截图移动到 artifacts 子目录。

### 6.2 后续建议

- 为 YAML 增加独立 lint 命令，不连接设备也能检查配置。
- 为新增模块提供脚手架，自动生成入口脚本和 YAML 模板。
- 建立模块注册机制，减少新增模块时手动复制文件。
- 建立 selector 覆盖率检查，避免模块配置重新引入坐标点击。
- 如模板后续增加列，再考虑给钉钉表格补充执行时间、设备号、模块名和错误摘要。
- 为 YAML 加载、钉钉表格导出和报告生成补充单元测试。
- 后续可按风险引入 OCR 文案识别，但应先保证截图链路稳定。

## 7. 验收标准

- 任一模块可通过对应入口脚本独立执行。
- 配置错误能在执行前输出清晰提示。
- 执行成功后，产物目录中存在截图、报告和 Poco 节点树。
- 钉钉在线表格中能看到按步骤排序的截图记录。
- 报告能展示截图总数和每步定位方式。
- 辅助工具不会在项目根目录遗留临时截图。
